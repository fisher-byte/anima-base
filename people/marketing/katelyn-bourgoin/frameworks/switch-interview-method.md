# Switch Interview Method

> **核心主张**: 通过理解用户从旧方案"切换"到新方案的完整时间线,发现真实购买动机

## 方法概述

Switch Interview 是基于 Bob Moesta 的 Jobs-to-be-Done 理论的深度访谈方法,专注于理解用户**切换行为**的完整故事。

**核心洞察**: 用户不是"购买产品",而是"雇佣产品"完成任务,并"解雇"旧方案。

## 理论基础

### Push & Pull 动力学

```
[旧方案] ← 推力(Push) | 拉力(Pull) → [新方案]
           焦虑/惯性   |  吸引力/习惯
```

**推力(Push)**: 促使离开旧方案的因素
- 旧方案失败的时刻
- 情绪触发(沮丧/焦虑)
- 外部压力(老板要求/截止日期)

**拉力(Pull)**: 吸引到新方案的因素
- 新方案的承诺
- 他人推荐
- 成功案例

**切换时刻**: 当 Push > Pull + 切换成本时

## Switch Interview 结构

### 时间线映射

```
第一次想到 → 被动寻找 → 主动搜索 → 决策 → 使用 → 推荐
   (First Thought)  (Passive)   (Active)  (Decision) (Use)  (Advocate)
```

### 核心问题框架

#### 阶段 1: 旧方案(Before)
**目标**: 理解旧方案和痛点

**问题**:
1. "在购买之前,你是如何解决这个问题的?"
2. "那个方案哪里不够理想?"
3. "你用了多久那个方案?"
4. "为什么当时没有换?"(理解惯性)

#### 阶段 2: 触发事件(Trigger)
**目标**: 识别"第一次想到要换"的时刻

**问题**:
1. "是什么让你开始考虑换方案?"
2. "那是什么时候发生的?"
3. "你当时在做什么?"
4. "你当时感觉如何?"

**关键**: 找到具体的触发事件,而非模糊的"一直想换"

#### 阶段 3: 搜索过程(Search)
**目标**: 理解评估和比较过程

**问题**:
1. "你是如何寻找新方案的?"
2. "你看了哪些其他选择?"
3. "为什么没选它们?"
4. "谁参与了决策?"
5. "你最担心什么?"(理解焦虑)

#### 阶段 4: 决策时刻(Decision)
**目标**: 理解最终决策因素

**问题**:
1. "是什么让你最终选择我们?"
2. "有没有差点不买的时刻?什么改变了你的想法?"
3. "你是如何说服自己/团队的?"

#### 阶段 5: 使用体验(Use)
**目标**: 验证价值实现

**问题**:
1. "使用后和预期有什么不同?"
2. "什么让你惊喜/失望?"
3. "你会如何向他人描述我们?"

## 访谈技巧

### 1. 开放式问题
**原则**: 让用户讲故事,而非回答是非题

❌ **封闭式**: "你喜欢这个功能吗?"
✅ **开放式**: "你是如何使用这个功能的?"

### 2. 深挖细节
**技巧**: 持续追问"然后呢?""你当时怎么想?"

**示例对话**:
```
Q: "你是如何发现我们的?"
A: "Google 搜索"
Q: "你搜了什么?"
A: "项目管理工具"
Q: "为什么那天搜索?"
A: "项目延期,老板问责"
Q: "能详细说说那次经历吗?"
[深挖到真实触发器]
```

### 3. 沉默的力量
**原则**: 问完问题后保持沉默,让用户思考

**避免**:
- 立即补充问题
- 提供答案选项
- 打断用户思考

### 4. 记录原话
**原则**: 逐字记录关键词汇,后续用于内容创作

**重点记录**:
- 情绪词汇("沮丧""焦虑""放心")
- 痛点描述(用户的原话)
- 理想结果表达

## 实施流程

### 准备阶段(1-2 天)

**1. 招募对象**
- 目标: 最近 2-4 周购买的客户
- 数量: 5-10 人
- 奖励: $50-100 礼品卡

**2. 准备脚本**
- 基于时间线结构
- 开放式问题
- 准备追问

**3. 技术设置**
- 视频通话工具(Zoom/Google Meet)
- 录音许可
- 转录工具(Otter.ai/Rev)

### 执行阶段(30-45 分钟/访谈)

**时间分配**:
- 开场(5 min): 建立信任,说明目的
- 旧方案(5 min): 理解背景
- 触发与搜索(15 min): 核心部分,深挖细节
- 决策与使用(10 min): 验证价值
- 收尾(5 min): 开放问题,感谢

**主持技巧**:
- 80/20 法则: 用户说 80%,你说 20%
- 好奇心 > 验证: 学习而非确认假设
- 具体 > 抽象: 追问具体例子

### 分析阶段(2-3 小时)

**1. 转录整理**
- 完整转录或标注关键段落
- 高亮情绪词汇和触发器

**2. 时间线映射**
- 绘制每个用户的切换时间线
- 标注关键时刻

**3. 模式识别**
- 寻找重复出现的触发器(3+ 次)
- 识别共同的决策因素
- 提炼核心洞察

**4. 洞察提炼**
- 使用"当...时,用户会..."公式
- 连接触发器→情绪→行为

## 常见错误

### ❌ 错误 1: 推销产品
**表现**: "我们在做 XX 产品,你觉得怎么样?"

**问题**: 用户给礼貌性答案,无法获得真实洞察

**正确**: "你上次遇到这个问题,是如何解决的?"

### ❌ 错误 2: 问假设性问题
**表现**: "如果我们加上 XX 功能,你会用吗?"

**问题**: 假设性答案不可靠

**正确**: "你上次想要这个功能,是什么时候?当时发生了什么?"

### ❌ 错误 3: 接受表面答案
**表现**:
```
Q: "你为什么买我们的产品?"
A: "因为功能好"
[停止追问]
```

**正确**:
```
A: "因为功能好"
Q: "哪个功能?"
Q: "你什么时候意识到需要这个功能?"
Q: "在那之前你怎么做?"
[持续深挖]
```

## 洞察应用

### 1. 产品开发
**从访谈中识别**:
- 旧方案的缺陷 → 产品改进方向
- 切换障碍 → 入门体验优化
- 惊喜时刻 → 核心价值强化

### 2. 营销文案
**直接使用用户原话**:
- 痛点描述 → Landing Page 开头
- 触发场景 → 广告文案
- 决策因素 → Feature 说明

### 3. 销售策略
**优化销售流程**:
- 识别高意向信号(特定触发器)
- 理解决策时间线
- 预判和消除顾虑

## 成功案例

### 案例: SaaS 项目管理工具

**访谈发现**:
- 5/7 用户的触发器: "项目延期被老板问责"
- 情绪: 焦虑、紧迫、害怕失去信任
- 旧方案: Excel/Email(手动、不透明)
- 决策因素: 快速上手 + 自动报告

**应用**:
- Landing Page 改为: "再也不用在周一早会上解释项目为什么延期"
- 强化功能: 一键生成进度报告
- 免费试用: 2 周,无需信用卡(降低切换成本)

**结果**:
- 转化率提升 160%
- 平均合同金额上升 30%

## 工具与资源

### 访谈脚本模板
```
开场:
"感谢你的时间。我想了解你购买之前的故事,
这不是推销,纯粹为了学习。"

核心问题:
1. "在购买之前,你是如何解决 [问题]?"
2. "是什么让你开始考虑换方案?"
3. "能详细说说那个时刻吗?"
4. "你是如何搜索的?看了哪些其他选择?"
5. "是什么让你最终选择我们?"
6. "使用后有什么变化?"

收尾:
"还有什么我应该问但没问的?"
```

### 分析模板
| 用户 | 旧方案 | 触发器 | 情绪 | 搜索过程 | 决策因素 |
|------|--------|--------|------|----------|----------|
| Sarah | Excel | 项目延期 | 焦虑 | Google搜索 | 快速上手 |
| John | Trello | 老板要求 | 紧迫 | 同事推荐 | 自动报告 |
| ... | ... | ... | ... | ... | ... |

**模式识别**: 多数用户因"问责压力"触发,需要"快速可见的价值"

## 关键要点

1. **行为 > 意见**: 关注用户做了什么,而非说想要什么
2. **时间线思维**: 完整映射切换过程,而非孤立的决策时刻
3. **情绪洞察**: 情绪触发购买,理性支持决策
4. **持续追问**: 表面答案下总有更深层的故事
5. **原话力量**: 用户的词汇比营销术语更有力
