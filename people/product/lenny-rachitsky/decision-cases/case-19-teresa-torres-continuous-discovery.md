# Case 19: Teresa Torres持续发现 — 每周与用户对话的纪律

> **决策者**: Teresa Torres (用户研究专家)  
> **案例来源**: Lenny's Podcast + 《Continuous Discovery Habits》  
> **决策类型**: 用户研究方法论  
> **证据等级**: A级（多公司实践验证）  
> **影响**: 用户研究标准流程

---

## 核心问题

**传统用户研究的问题**:
```
问题1: 研究与开发脱节
- 研究团队做调研
- 扔给产品团队一份报告
- 产品团队不理解/不采纳

问题2: 频率太低
- 每季度做一次大调研
- 用户需求已变化
- 决策基于过时信息

问题3: 样本偏差
- 只找愿意参与的用户
- 忽略沉默的大多数
```

**Teresa的解决方案**: Continuous Discovery

---

## 核心框架

### 持续发现习惯

**定义**: 产品团队每周至少与3个用户对话

```
频率：每周（而非每季度）
参与者：整个产品团队（而非只研究员）
目标：理解用户需求（而非验证想法）
```

### Opportunity Solution Tree（机会方案树）

**结构**:
```
Outcome（目标）
  └─ Opportunity（用户痛点/机会）
       └─ Solution（解决方案）
            └─ Experiment（实验）
```

**使用方法**:
```
Step 1: 定义Outcome
  - 想达成什么业务目标？
  - 示例："提升新用户7日留存至60%"

Step 2: 识别Opportunity
  - 用户访谈发现痛点
  - 示例："新用户不知道如何开始"

Step 3: 生成Solution
  - 针对痛点设计方案
  - 示例："引导式Onboarding"

Step 4: 实验验证
  - MVP测试
  - 示例："A/B测试Onboarding流程"
```

---

## 决策启发式

### 1. 每周3次用户对话

```
IF 产品团队
THEN 每周至少3次用户对话：
  - 周一：1次
  - 周三：1次
  - 周五：1次

目标：持续理解用户，而非季度大调研
```

### 2. 整个团队参与

```
IF 用户访谈
THEN 整个产品团队（PM/设计/工程）都参与：
  - 不是只有研究员听
  - 全员理解用户（更有同理心）
  - 决策基于一手信息
```

### 3. 机会驱动（而非方案驱动）

```
传统流程（方案驱动）:
1. 想出方案
2. 找用户验证 ❌

正确流程（机会驱动）:
1. 发现用户痛点（Opportunity）
2. 针对痛点设计方案（Solution）✅
```

---

**案例总结**:
- 决策: 建立持续发现习惯（每周3次用户对话）
- 工具: Opportunity Solution Tree
- 收益: 决策基于一手用户洞察，产品契合度提升
- 启发式: 频率>深度，每周3次>每季度1次深度调研
- 适用: 所有产品团队
