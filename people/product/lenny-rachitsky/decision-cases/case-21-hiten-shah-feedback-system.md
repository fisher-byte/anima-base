# Case 21: Hiten Shah客户反馈系统 — 如何系统化收集用户声音

> **决策者**: Hiten Shah (FYI/Product Habits创始人)  
> **案例来源**: Lenny's Podcast + Hiten博客  
> **决策类型**: 用户反馈机制  
> **证据等级**: A级（多产品实践）  
> **影响**: 客户反馈收集标准流程

---

## 核心问题

**传统反馈收集的问题**:
```
问题1: 反馈分散
- Email/Slack/Twitter/Support各处都有
- 无法系统化整理
- 洞察淹没在噪音中

问题2: 无法量化
- "很多用户要求X功能"（多少是很多？）
- 凭感觉决策
- 容易偏向"声音大"的用户

问题3: 缺乏跟进
- 用户提反馈
- 团队记录但不行动
- 用户失望，不再反馈
```

**Hiten的解决方案**: 系统化反馈流程

---

## 核心框架

### 3层反馈系统

#### Layer 1: 收集（Collect）

**多渠道统一收集**:
```
反馈来源：
- 用户访谈
- Support tickets
- Email
- 产品内反馈按钮
- NPS调查
- 社交媒体

关键：统一到一个系统（如Productboard/Canny）
```

#### Layer 2: 分类（Categorize）

**标准化分类**:
```
维度1: 功能类别
- 示例：Onboarding/报表/集成/性能

维度2: 用户角色
- 示例：免费用户/付费用户/企业客户

维度3: 优先级
- P0: 影响核心功能
- P1: 重要但非紧急
- P2: Nice to have

维度4: 数量统计
- 多少用户提到这个问题？
- 用数据说话
```

#### Layer 3: 行动（Act）

**闭环反馈**:
```
Step 1: 决策（做/不做/延后）
Step 2: 通知用户决策结果
Step 3: 做了后再次通知

案例：
- 用户提："希望有暗黑模式"
- 团队决策："Q3做"
- Email用户："您的建议已排期，Q3上线"
- Q3上线后："暗黑模式已上线，感谢您的建议"

结果：用户感受到被重视 → 持续反馈
```

---

## 决策启发式

### 1. 量化反馈

```
IF 收到功能请求
THEN 追踪统计：
  - 多少用户提到？
  - 什么类型用户？（免费/付费/企业）
  - 紧急程度？

决策依据：数据而非感觉
```

### 2. 闭环沟通

```
IF 用户提反馈
THEN 必须回复：
  - 即使不做，也要说明原因
  - 做了，要通知用户

避免：静默处理（用户以为被忽视）
```

### 3. 区分痛点 vs 方案

```
用户常说方案（而非痛点）：
- 用户说："我要X功能"（方案）
- 应该问："您想解决什么问题？"（痛点）

IF 理解痛点
THEN 可能有更好的方案（而非用户要求的）
```

---

**案例总结**:
- 决策: 建立系统化反馈收集流程
- 方法: 收集→分类→行动（闭环）
- 工具: Productboard/Canny统一管理
- 启发式: 量化反馈+闭环沟通+区分痛点vs方案
- 适用: 所有产品团队
