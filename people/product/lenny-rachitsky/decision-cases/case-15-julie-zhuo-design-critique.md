# Case 15: Julie Zhuo设计评审方法 — 如何做高效设计critique

> **决策者**: Julie Zhuo (前Facebook设计VP)  
> **案例来源**: Lenny's Podcast + 《The Making of a Manager》  
> **决策类型**: 团队协作方法论  
> **证据等级**: A级（Facebook实践经验）  
> **影响**: 科技公司设计评审标准流程

---

## 核心问题

**传统设计评审的问题**:
- 评审变成"批评大会"
- 设计师感到被攻击
- 反馈不actionable
- 会议效率低

**Julie的解决方案**: 结构化评审框架

---

## 决策框架

### 评审前准备（设计师）

**必须明确3件事**:
1. **目标是什么？**（解决什么问题？）
2. **当前状态？**（到达哪个阶段？）
3. **需要什么反馈？**（具体问题是什么？）

**案例**:
```
❌ 差的准备："大家看看这个设计"
✅ 好的准备："我们要解决新用户激活问题，这是第二轮迭代，
   我不确定CTA按钮的颜色和位置，希望大家重点给这方面反馈"
```

### 评审中框架（参与者）

**Julie的"梯子法则"**:
```
Level 1: 我喜欢/不喜欢 ❌
  - 主观感受
  - 无法执行

Level 2: 这里有问题 ⚠️
  - 指出问题
  - 但没有解决方案

Level 3: 这里有问题，因为... ✅
  - 指出问题
  - 解释原因
  - 可讨论

Level 4: 这里有问题，因为...，建议... ✅✅
  - 指出问题
  - 解释原因
  - 提供方案
  - 最有价值

目标：尽量达到Level 3-4
```

**实际应用**:
```
❌ Level 1："我不喜欢这个蓝色"
❌ Level 2："这个按钮太小了"
✅ Level 3："这个按钮太小，因为用户可能误触，
   参考iOS人机界面指南建议最小44x44pt"
✅✅ Level 4："这个按钮太小，因为用户可能误触，
   建议增大到44x44pt，或者增加周围空白区域"
```

---

## 决策启发式

### 1. 评审前明确3W

```
IF 召开设计评审
THEN 设计师必须提前说明：
  - What: 解决什么问题？
  - Where: 当前在哪个阶段？（概念/原型/定稿）
  - Want: 需要什么类型反馈？

这样团队才能给有针对性的反馈
```

### 2. 梯子法则（Level 3+）

```
IF 给设计反馈
THEN 至少达到Level 3：
  - 指出具体问题
  - 解释为什么是问题
  - （最好）提供建议方案

避免Level 1-2的主观评价
```

### 3. 区分阶段反馈

```
IF 早期概念阶段
THEN 反馈方向/思路（不纠结细节）

IF 定稿阶段  
THEN 反馈执行细节（像素级）

不同阶段，反馈粒度不同
```

---

**案例总结**:
- 决策: 建立结构化设计评审流程
- 框架: 3W准备 + 梯子法则
- 收益: 评审效率提升，设计师体验改善
- 启发式: 反馈必须actionable（Level 3+）
