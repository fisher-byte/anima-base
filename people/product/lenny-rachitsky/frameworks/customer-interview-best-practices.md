# Customer Interview Best Practices

**来源：** Lenny's Podcast featuring Teresa Torres (2022-06-30)  
**作者：** Lenny Rachitsky & Teresa Torres  
**适用场景：** 用户访谈、需求发现、产品验证、持续探索

---

## 核心理念

> "The most common mistake in customer interviews is asking customers what they want. The goal is to understand their behavior, not collect feature requests."  
> —Teresa Torres

**关键洞察：**
- 好的访谈**揭示行为**，而非收集意见
- 最佳频率：**每周**与客户对话（Continuous Discovery）
- 访谈不是验证假设，而是**发现机会**

---

## 框架结构

### 1. 访谈准备：Before the Interview

#### A. 明确研究目标

**三种访谈类型：**

| 类型 | 目标 | 时机 | 问题示例 |
|------|------|------|---------|
| **探索式** | 发现问题和机会 | 产品规划期 | "描述上次使用XX的经历" |
| **验证式** | 测试假设和方案 | 原型设计期 | "这个方案能解决你的问题吗？" |
| **生成式** | 激发新想法 | 创新阶段 | "理想情况下，这应该如何工作？" |

**目标设定模板：**
```
我们想要理解：[用户群体]在[特定场景]下的[行为/痛点/目标]
为了：[做出什么产品决策]
```

示例：
- "理解设计师在协作时的沟通障碍，为了设计更好的评论功能"
- "发现用户取消订阅的真实原因，为了降低流失率"

#### B. 招募合适的参与者

**招募标准：**
1. ✅ **代表目标用户**：符合用户画像
2. ✅ **有真实经历**：最近使用过产品
3. ✅ **愿意分享**：不是只为奖励来的
4. ✅ **多样性**：不同使用场景、成熟度

**招募渠道：**
- 产品内邀请（最佳）
- 客服推荐
- 社区招募
- 第三方平台（UserTesting, Respondent）

**样本量：**
- **探索式**：5-8人即可发现大部分问题
- **验证式**：10-15人获得模式确认
- **持续访谈**：每周2-3人

#### C. 设计访谈指南

**结构模板（60分钟）：**

```
1. 开场（5分钟）
   - 自我介绍
   - 说明目的和流程
   - 征得录音许可

2. 破冰（5分钟）
   - 了解背景："告诉我你的工作"
   - 建立信任

3. 主体部分（40分钟）
   - 行为回顾：描述具体经历
   - 痛点挖掘：困难和挫折
   - 目标理解：期望达成什么

4. 收尾（10分钟）
   - 开放问题："还有什么想分享的？"
   - 感谢和后续
```

### 2. 访谈技巧：During the Interview

#### A. 核心原则：5个要做 & 5个不要做

**✅ 5个要做（DO）**

**1. 问行为，不问意见**
- ❌ "你觉得这个功能有用吗？"
- ✅ "上次你遇到这个问题时，你是怎么做的？"

**2. 追问"为什么"（5 Whys）**
```
用户："我讨厌这个界面"
你："为什么？"
用户："太慢了"
你："慢体现在哪里？"
用户："加载时间超过3秒"
你："3秒为什么对你来说太久？"
用户："因为我的客户在旁边等"
```

**3. 要求讲故事（Story-Based Questions）**
- ✅ "描述上次你使用XX的完整过程"
- ✅ "带我走一遍你的工作流程"
- ✅ "讲讲最近一次遇到这个问题的经历"

**4. 观察非语言信号**
- 犹豫、停顿 → 可能在掩饰真相
- 兴奋、快速 → 触及痛点
- 重复某个词 → 关键概念

**5. 保持中立，不引导**
- ❌ "你也觉得XX很烦吧？"
- ✅ "你对XX有什么想法？"

**❌ 5个不要做（DON'T）**

**1. 不要问假设性问题**
- ❌ "如果有XX功能，你会用吗？"
- ✅ "上次没有XX时，你是怎么解决的？"

**2. 不要接受模糊答案**
- 用户："有时候不太好用"
- ❌ 接受并继续
- ✅ "能举个具体例子吗？"

**3. 不要提前展示方案**
- ❌ 先展示原型，再问"你觉得怎么样？"
- ✅ 先理解问题，最后再验证方案

**4. 不要害怕沉默**
- 沉默是思考的信号
- 等待3-5秒，不要立即填补

**5. 不要一次问多个问题**
- ❌ "你为什么用这个工具？觉得哪里好？哪里不好？"
- ✅ 一次一个，等回答完再继续

#### B. 高级技巧：4个经典问题模式

**1. Opportunity Questions（机会挖掘）**
```
"描述上次你[目标行为]失败的经历"
"什么阻止了你完成[任务]？"
"如果可以改变一件事，你会改什么？"
```

**2. Behavior Questions（行为探索）**
```
"带我走一遍上次你[具体行为]的过程"
"你多久做一次[行为]？"
"上次是什么时候？发生了什么？"
```

**3. Contrast Questions（对比理解）**
```
"这个和你之前用的XX有什么不同？"
"为什么选择A而不是B？"
"成功和失败的情况有什么区别？"
```

**4. Laddering Questions（深度挖掘）**
```
"为什么这对你很重要？"
"如果没有XX会怎样？"
"这个问题解决了会给你带来什么？"
```

### 3. 常见访谈场景

#### 场景1：PMF验证访谈

**核心问题（必问）：**

1. **痛点验证**
   - "描述上次遇到这个问题的经历"
   - "这个问题有多严重？（1-10分）"
   - "你现在如何解决？"

2. **解决方案契合度**
   - "我们的产品解决了你的问题吗？"
   - "如果明天我们消失，你会怎么办？"（Sean Ellis PMF问题）
   - "你会推荐给朋友吗？为什么？"

3. **愿付价格**
   - "你现在为类似方案付多少钱？"
   - "你愿意为这个付多少？"
   - "什么价格你会觉得太贵？太便宜（不可信）？"

#### 场景2：功能优先级访谈

**结构化方法：Job-To-Be-Done访谈**

```
1. 情境设定
   "描述上次你需要[完成某任务]的场景"

2. 痛点挖掘
   "过程中遇到了什么困难？"
   "什么让你感到挫败？"

3. 期望结果
   "理想情况下，这应该如何完成？"
   "什么算成功？什么算失败？"

4. 权衡取舍
   "如果只能改善一件事，你会选什么？"
```

#### 场景3：流失原因访谈

**敏感话题处理：**

1. **建立信任**
   - "我们想变得更好，你的反馈很重要"
   - "我不是来说服你留下的，只是想理解"

2. **核心问题**
   - "什么让你决定停止使用？"
   - "具体发生了什么事？"
   - "你转向了什么替代方案？"

3. **深度挖掘**
   - "如果我们改变XX，你会考虑回来吗？"
   - "还有什么其他原因吗？"

### 4. 持续访谈（Continuous Discovery）

#### Teresa Torres的每周访谈法

**为什么每周访谈？**
- ✅ 建立肌肉记忆，访谈技巧提升
- ✅ 持续发现新机会，而非季度突击
- ✅ 团队共同参与，统一理解

**实施步骤：**

**Step 1: 自动化招募（Week 0）**
```
设置：
- 产品内NPS调查后自动邀请
- 奖励：$50礼品卡或产品积分
- 目标：每周2-3个访谈
```

**Step 2: 团队共同参与（每周）**
```
参与者：
- PM主持
- 设计师、工程师旁听
- 实时记录到Miro/FigJam

注意：
- 最多3人参与（避免压迫感）
- 轮流主持（练习技能）
```

**Step 3: 立即分析（访谈后30分钟）**
```
团队讨论：
1. "我们学到了什么？"
2. "哪些是新发现？哪些是验证？"
3. "这改变了我们的优先级吗？"
4. 更新Opportunity Solution Tree
```

**Step 4: 知识沉淀（每周五）**
```
文档化：
- 更新用户画像
- 记录痛点库
- 归档访谈记录
- 分享给全公司
```

## 分析与应用

### 从访谈到洞察：4步分析法

**Step 1: 主题提取（Thematic Analysis）**
```
方法：
1. 逐字稿分段标注
2. 提取关键引用
3. 识别重复模式
4. 归类主题

工具：
- Dovetail（自动化标注）
- Miro（可视化分组）
- Notion（结构化记录）
```

**Step 2: 痛点优先级排序**
```
评估维度：
- 频率：多少用户提到？
- 严重性：1-10分
- 紧迫性：多快需要解决？
- 可解决性：我们能解决吗？
```

**Step 3: 机会映射（Opportunity Mapping）**
```
使用Opportunity Solution Tree：
- Outcome（目标）
  ↓
- Opportunities（机会）
  ↓
- Solutions（方案）
  ↓
- Experiments（实验）
```

**Step 4: 行动清单**
```
输出：
- 立即行动：下周就做
- 短期计划：本季度
- 长期愿景：明年
- 不做清单：为什么不做
```

## 常见错误与解决方案

### ❌ 错误1：验证偏见（Confirmation Bias）

**表现：**
- 只问能支持自己假设的问题
- 忽略反对信号
- 选择性记录

**解决：**
- ✅ 预设假设，但主动寻找反证
- ✅ 问"什么情况下你不会用？"
- ✅ 团队多人参与，交叉验证

### ❌ 错误2：问未来，不问过去

**错误示例：**
- "你会用这个功能吗？" → 90%说会，实际10%会用

**正确方式：**
- ✅ "上次你是怎么解决这个问题的？"
- ✅ 观察真实行为，而非假设意图

### ❌ 错误3：没有足够的追问

**浅层：**
- 用户："这个不太好用"
- PM："好的，我记下了" ❌

**深度：**
- 用户："这个不太好用"
- PM："能具体说说哪里不好用吗？"
- 用户："加载很慢"
- PM："多慢算慢？上次等了多久？"
- 用户："大概5秒，但我在开会时用，客户在等"
- PM："所以是在会议场景下的时间压力？" ✅

## 实施工具包

### 访谈脚本模板

```markdown
# 客户访谈脚本

**目标：** [理解XX行为/验证XX假设]
**时间：** 60分钟
**参与者：** [用户画像]

## 开场（5分钟）
- 自我介绍："我是XX，负责XX产品"
- 说明目的："我们想了解您如何使用XX"
- 征得许可："可以录音吗？会保密"

## 背景（10分钟）
1. 告诉我你的工作/角色
2. 你多久做一次[目标任务]？
3. 你目前如何完成[任务]？

## 深度探索（35分钟）
4. 描述上次你[行为]的完整过程
5. 遇到了什么困难？
6. 理想情况下应该如何？
7. [方案演示]这个能解决你的问题吗？
8. 什么是必须有的？什么是nice-to-have？

## 收尾（10分钟）
9. 还有什么想分享的？
10. 可以再找你聊吗？
11. 感谢 + 礼品卡

## 记录要点
- [ ] 核心痛点
- [ ] 现有解决方案
- [ ] 购买意愿/价格敏感度
- [ ] 金句引用
```

## 关键要点

1. **问行为，不问意见**：过去的行为比未来的承诺更可靠
2. **持续进行，不是一次性**：每周2-3次访谈，建立持续探索习惯
3. **团队共同参与**：PM、设计师、工程师一起访谈，统一理解
4. **深度追问**：5 Whys原则，挖掘真实动机
5. **立即分析**：访谈后30分钟内团队讨论，趁记忆新鲜
