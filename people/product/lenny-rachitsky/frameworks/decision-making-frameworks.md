## 概述

Lenny Rachitsky 收集整理的决策框架合集，帮助团队在面对复杂决策时减少争论、提升质量、加快速度。

**核心理念:** 好的决策框架能让团队用更少的时间做出更好的决策。

## 主要决策框架

### 1. DACI Framework
**用途:** 明确决策角色和流程

**角色定义:**
- **D**river (驱动者): 负责推动决策流程
- **A**pprover (批准者): 最终决策者（通常只有 1 人）
- **C**ontributor (贡献者): 提供意见和输入
- **I**nformed (知情者): 需要被告知决策结果

**何时使用:**
- 重大决策
- 涉及多个团队
- 需要明确责任

**关键原则:**
- Approver 只有 1 人（避免委员会决策）
- Driver 推动，但不一定是决策者
- 区分"咨询"和"知会"

**案例:**
```
决策: 是否推出新功能 X

Driver: 产品经理 Alice
Approver: VP Product Bob
Contributors: 工程、设计、数据团队
Informed: 全公司
```

### 2. Two-Way vs One-Way Door Decisions
**来源:** Jeff Bezos (Amazon)

**分类:**
- **One-Way Door（单向门）**: 不可逆决策，需要慎重
  - 例: 公司并购、关键人员招聘、战略方向
  - 方法: 深思熟虑，收集充分信息，高层决策

- **Two-Way Door（双向门）**: 可逆决策，可以快速尝试
  - 例: 功能实验、设计调整、流程优化
  - 方法: 快速决策，快速实验，快速迭代

**关键洞察:**
- 大多数决策是 Two-Way Door
- 不要用 One-Way Door 的流程处理 Two-Way Door
- 提升决策速度的关键是识别哪些是可逆的

### 3. Disagree and Commit
**来源:** Amazon 领导力原则

**适用场景:**
- 团队有分歧
- 需要快速推进
- 成本可控

**流程:**
1. 充分讨论，表达不同意见
2. 决策者做出决定
3. 所有人全力支持执行（即使不同意）
4. 快速验证，必要时调整

**关键原则:**
- 不是妥协，是信任
- 反对后必须全力支持
- 设置明确的验证点

### 4. Pre-Mortem（事前验尸）
**用途:** 在决策执行前识别风险

**流程:**
1. 假设项目已经失败
2. 每人写下"失败原因"
3. 分享讨论所有原因
4. 针对主要风险制定预案

**优点:**
- 鼓励批判性思考
- 心理安全（假设性场景）
- 提前识别盲点

**何时使用:**
- 重大项目启动前
- 策略决策前
- 风险较高的决定

### 5. Bets Framework（下注框架）
**来源:** Basecamp / 37signals

**核心思想:** 把项目当作"赌注"来管理

**要素:**
- **Bet Size（赌注大小）**: 6周 vs 2周项目
- **Appetite（胃口）**: 愿意投入多少资源
- **Pitch（推介）**: 清晰的项目提案
- **No Default Yes（默认不是）**: 不做比做更容易

**决策标准:**
- 机会成本是什么？
- 这个赌注值得吗？
- 我们愿意为此放弃什么？

### 6. ICE Scoring（优先级评分）
**用途:** 项目/功能优先级排序

**评分维度:**
- **I**mpact（影响）: 1-10 分
- **C**onfidence（信心）: 0-100%
- **E**ase（容易度）: 1-10 分

**计算:**
```
ICE Score = (Impact × Confidence × Ease) / 100
```

**优点:**
- 简单直观
- 强制量化思考
- 易于比较

**何时使用:**
- Feature prioritization
- Roadmap planning
- 资源分配

### 7. RICE Scoring（增强版 ICE）
**来源:** Intercom

**评分维度:**
- **R**each（触达）: 影响多少用户
- **I**mpact（影响）: 1-3 分（0.25 / 0.5 / 1 / 2 / 3）
- **C**onfidence（信心）: 0-100%
- **E**ffort（成本）: 人月

**计算:**
```
RICE Score = (Reach × Impact × Confidence) / Effort
```

**优点:**
- 更全面
- 考虑成本
- 适合资源受限场景

### 8. Confidence Levels（信心等级）
**来源:** Shishir Mehrotra (Coda)

**用途:** 表达决策信心，减少争论

**等级:**
- **🟢 Strong Confidence (8-10)**: 强烈建议
- **🟡 Medium Confidence (5-7)**: 倾向但开放
- **🔴 Low Confidence (1-4)**: 不确定

**使用方式:**
```
"我认为应该做 X (信心等级: 7/10)"
而不是
"我认为应该做 X"（听起来像绝对确定）
```

**优点:**
- 鼓励观点表达
- 减少极化讨论
- 明确不确定性

### 9. Regret Minimization Framework
**来源:** Jeff Bezos

**核心问题:**
"当我 80 岁回顾时，我会后悔这个决定吗？"

**适用:**
- 职业选择
- 重大战略
- 长期方向

**关键:**
- 关注长期，而非短期得失
- 考虑后悔成本
- 偏向行动而非观望

## 如何选择合适的框架

### 按决策类型

| 决策类型 | 推荐框架 |
|---------|---------|
| 角色不清 | DACI |
| 优先级排序 | ICE / RICE |
| 风险识别 | Pre-Mortem |
| 快速决策 | Two-Way Door |
| 团队分歧 | Disagree and Commit |
| 表达不确定 | Confidence Levels |
| 资源分配 | Bets Framework |
| 长期战略 | Regret Minimization |

### 按团队阶段

**早期（< 10人）:**
- Two-Way Door
- Disagree and Commit

**成长期（10-50人）:**
- DACI
- ICE Scoring
- Pre-Mortem

**成熟期（50+人）:**
- 所有框架
- 需要更正式的流程

## 实施建议

### 1. 不要教条
- 框架是工具，不是规则
- 根据情况灵活调整
- 可以混合使用

### 2. 从简单开始
- 先试 1-2 个框架
- 团队熟悉后再引入新的
- 避免"框架疲劳"

### 3. 明确何时使用
- 为不同决策类型指定框架
- 写入团队文档
- 定期回顾优化

### 4. 重视执行
- 最好的框架是被使用的框架
- 决策后快速行动
- 设置验证点和回顾

## 常见陷阱

### 1. 分析瘫痪
❌ 过度使用框架，决策变慢
✅ 区分重大和日常决策

### 2. 伪精确
❌ 过度相信分数和公式
✅ 定量 + 定性结合

### 3. 委员会决策
❌ 所有人都是 Approver
✅ 明确单一决策者

### 4. 忽视反馈
❌ 决策后不回顾
✅ 定期复盘决策质量
