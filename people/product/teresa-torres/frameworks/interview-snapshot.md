# Interview Snapshot

**Interview Snapshot** 是 Teresa Torres 开发的访谈综合工具，帮助团队快速记录和分享从单次用户访谈中学到的内容。

## 框架概述

当你开始每周访谈用户时，很容易被大量信息淹没。Interview Snapshot 提供了一个结构化的方式来记录关键发现。

### 三个关键工具

Teresa Torres 推荐团队使用三个不同的工具来追踪访谈学习：

| 工具 | 用途 | 更新频率 |
|------|------|----------|
| **Interview Snapshot** | 总结单次访谈的发现 | 每次访谈后 |
| **Experience Map** | 可视化用户旅程中的所有机会 | 访谈积累后 |
| **Opportunity Solution Tree** | 管理整体发现过程 | 持续更新 |

## Interview Snapshot 结构

### 基本模板

```markdown
# Interview Snapshot: [用户名/编号]
**日期**: YYYY-MM-DD
**访谈者**: [团队成员]
**时长**: XX 分钟

## 用户背景
- 角色/职位
- 使用产品的时间
- 关键背景信息

## 故事摘要
[描述用户分享的具体故事]

## 体验地图（Experience Map）
1. [步骤1] → 痛点/机会
2. [步骤2] → 痛点/机会
3. [步骤3] → 痛点/机会
...

## 发现的机会
- [ ] 机会1: [用户语言描述的需求/痛点]
- [ ] 机会2: [用户语言描述的需求/痛点]
- [ ] 机会3: [用户语言描述的需求/痛点]

## 快速事实（Quick Facts）
- [任何有用的数据点或引用]

## 后续行动
- [ ] 将机会添加到 OST
- [ ] 需要深入探索的领域
```

## 核心原则

### 1. 收集具体故事，而非观点

> "We want to collect specific stories about past behavior, not opinions about what they might do in the future."
> — Teresa Torres

**✓ 好的问题**:
- "上次你遇到这个问题时发生了什么？"
- "带我走一遍你完成这个任务的过程"

**✗ 避免的问题**:
- "你觉得这个功能怎么样？"
- "如果我们做X，你会用吗？"

### 2. 用用户的语言记录

- 不要翻译成产品术语
- 保留用户的原话
- 机会应该描述问题，不是解决方案

### 3. 即时综合

- 访谈后立即创建 Snapshot
- 趁记忆新鲜时记录
- Product Trio 共同完成

## 从故事中挖掘机会

### 故事挖掘技术（Story Excavation）

Teresa Torres 的 "Excavate the Story" 方法：

```markdown
引导问题：
1. "然后呢？" - 推动时间线
2. "那是什么样的？" - 获取细节
3. "你当时在想什么？" - 了解心理状态
4. "那有多重要？" - 评估优先级
5. "你还尝试过什么？" - 发现变通方案
```

### 机会识别信号

| 信号 | 示例 |
|------|------|
| 表达挫折 | "这真的很烦人..." |
| 变通方案 | "我通常会这样绕过它..." |
| 未满足需求 | "我希望能够..." |
| 时间浪费 | "我花了很长时间..." |
| 困惑 | "我不确定如何..." |

## 实施建议

### 谁应该参与

**Product Trio 共同访谈**:
- PM: 关注用户需求和业务影响
- Designer: 关注用户体验和痛点
- Engineer: 关注技术可行性和实现细节

### 访谈节奏

```markdown
推荐频率：
- 最少：每周 1 次用户访谈
- 理想：每周 2-3 次
- 团队一起参与，而非轮流
```

### 存储和共享

- 集中存储所有 Snapshots
- 使用统一命名规范
- 定期回顾和识别模式

## 与 Experience Map 的关系

Interview Snapshot 中的故事步骤会累积形成 Experience Map：

```
Interview 1: A → B → C → D
Interview 2: A → B → E → D
Interview 3: A → F → C → D
          ↓
Experience Map: 综合所有路径和机会
```

## 常见错误

1. **记录观点而非故事**
   - 错：用户说他们喜欢功能X
   - 对：用户描述了他们如何使用功能X完成任务

2. **过快总结**
   - 没有挖掘足够的细节
   - 错过重要的痛点

3. **单独完成**
   - PM 独自访谈和记录
   - 团队缺乏共同理解

4. **创建后遗忘**
   - 没有将发现连接到 OST
   - 没有追踪机会的演变
