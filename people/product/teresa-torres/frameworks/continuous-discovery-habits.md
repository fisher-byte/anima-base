# Continuous Discovery Habits

**Continuous Discovery Habits** 是 Teresa Torres 提出的产品发现方法论体系，强调将发现工作转化为团队的日常习惯，而非一次性活动。

## 框架概述

### 核心定义

> "Continuous discovery is, at a minimum, weekly touchpoints with customers by the team building the product, where they're conducting small research activities in pursuit of a desired outcome."
> — Teresa Torres

关键要素：
- **Weekly（每周）**: 不是每月或每季度，而是每周
- **By the team（由团队）**: 不只是研究员，整个 Product Trio
- **Small research activities（小型研究活动）**: 不需要大型研究项目
- **Desired outcome（期望结果）**: 有明确的目标

## 六大核心习惯

### Habit 1: 以结果为导向

```markdown
传统方式：
老板/利益相关者 → 功能需求 → 团队构建

持续发现方式：
公司战略 → 期望结果 → 团队探索 → 最佳解决方案
```

**关键实践**:
- 接受可衡量的结果，而非功能清单
- 结果应该是团队能够影响的
- 与公司战略明确对齐

### Habit 2: 每周用户接触

```markdown
目标：每周与用户的持续接触

形式：
- 用户访谈（首选）
- 用户观察
- 可用性测试
- 用户反馈回顾
```

**为什么每周**:
- 建立持续的用户理解
- 避免大型研究项目的开销
- 快速获得反馈循环

### Habit 3: Product Trio 协作

**Product Trio 组成**:
- **Product Manager**: 商业价值和用户需求
- **Designer**: 用户体验和可用性
- **Engineer**: 技术可行性和实现

**协作要求**:
- 三人一起参与用户访谈
- 共同做决策
- 共同承担责任

> "Discovery is a team sport, not an individual activity."

### Habit 4: 探索机会空间

使用 **Opportunity Solution Tree** 结构化探索：

```
           [Outcome]
               |
    +----------+----------+
    |          |          |
[机会1]    [机会2]    [机会3]
```

**关键实践**:
- 用用户语言表达机会
- 持续从访谈中添加新机会
- 结构化组织机会层级

### Habit 5: 比较和对比

> "Good product decisions are 'compare and contrast' decisions, not 'whether or not' decisions."

**实践方法**:
- 同时考虑 2-3 个解决方案
- 为每个方案设计假设测试
- 基于证据选择最佳方案

### Habit 6: 快速假设测试

```markdown
测试节奏：
- 每周至少 1 个假设测试
- 1-2 天内完成
- 快速迭代学习
```

**测试类型**:
- 用户访谈中的问题
- 纸面原型/线框图测试
- 烟雾测试
- 简单实验

## 实施路径

### 阶段 1: 建立基础（1-2 个月）

```markdown
□ 定义清晰的期望结果
□ 建立 Product Trio
□ 设立每周访谈机制
□ 创建第一棵 Opportunity Solution Tree
```

### 阶段 2: 建立节奏（2-4 个月）

```markdown
□ 每周进行用户访谈
□ 定期更新 OST
□ 开始假设测试
□ 建立 Interview Snapshot 习惯
```

### 阶段 3: 持续优化（4+ 个月）

```markdown
□ 优化访谈技巧
□ 加快测试节奏
□ 更好地平衡发现和交付
□ 扩展到更多团队
```

## 常见阻力和应对

### 阻力 1: "我们没时间"

**应对**:
- 从每周 30 分钟开始
- 将访谈整合到现有流程
- 证明短期投入的长期价值

### 阻力 2: "我们接触不到用户"

**应对**:
- 建立自动化招募渠道
- 利用现有接触点（客服、销售）
- 使用产品内的招募机制

### 阻力 3: "老板只给我们功能需求"

**应对**:
- 从小处开始，在功能中应用发现
- 展示发现带来的成果
- 逐步建立信任，获得更多自主权

## 与 Scrum/Agile 的结合

### Dual Track Development

```
发现轨道                    交付轨道
    |                           |
[用户访谈]               [Sprint 规划]
    ↓                           ↓
[机会发现]               [开发实现]
    ↓                           ↓
[解决方案设计]           [测试发布]
    ↓                           ↓
[假设测试]               [监测指标]
    |                           |
    +———— 信息流动 ————+
```

### 关键整合点

- **Sprint 规划**: 使用 OST 确定优先级
- **Backlog**: 用验证过的机会填充
- **Retro**: 回顾发现过程的有效性

## 成功指标

### 过程指标

| 指标 | 目标 |
|------|------|
| 每周用户访谈次数 | ≥ 1 |
| OST 更新频率 | 每周 |
| 假设测试频率 | 每周 ≥ 1 |
| Trio 参与率 | 100% |

### 结果指标

| 指标 | 意义 |
|------|------|
| 想法到验证的时间 | 发现效率 |
| 功能采用率 | 解决方案有效性 |
| 期望结果达成度 | 最终影响 |
