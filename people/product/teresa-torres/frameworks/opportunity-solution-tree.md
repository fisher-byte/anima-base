# Opportunity Solution Tree (OST)

**Opportunity Solution Tree** 是 Teresa Torres 开发的核心可视化框架，帮助产品团队从期望的结果出发，系统性地探索机会空间并选择最佳解决方案。

## 框架结构

OST 由四个层级组成，自上而下：

```
        [Desired Outcome]
              |
    +---------+---------+
    |         |         |
[Opportunity][Opportunity][Opportunity]
    |              |
+---+---+    +----+----+
|       |    |         |
[Sol.] [Sol.] [Sol.] [Sol.]
  |       |      |       |
[Exp.]  [Exp.] [Exp.]  [Exp.]
```

### 1. Desired Outcome（期望结果）
- 团队要驱动的业务指标
- 必须是可衡量的
- 应该与公司战略目标一致
- 例如：提高用户留存率、增加激活率

### 2. Opportunities（机会）
- 用户的需求、痛点和期望
- 来自于用户访谈和研究
- 应该用用户的语言表达
- 机会是问题空间，不是解决方案

### 3. Solutions（解决方案）
- 解决特定机会的产品想法
- 每个机会可以有多个解决方案
- 鼓励探索多种可能性

### 4. Experiments（实验）
- 用于验证解决方案假设的测试
- 快速、低成本的验证方式
- 帮助决定是否继续投入

## 核心原则

### Compare and Contrast（比较与对比）
> "Good product decisions are 'compare and contrast' decisions, not 'whether or not' decisions."
> — Teresa Torres

OST 强调同时考虑多个选项，而非逐个评估：
- 同时探索 2-3 个机会
- 每个机会准备 2-3 个解决方案
- 比较哪个最有希望，而非决定要不要做某一个

### Outcome-Driven（结果驱动）
- 从结果开始，而非从想法开始
- 所有决策都应追溯到期望结果
- 避免被"好点子"带偏

### Living Document（活文档）
- OST 随着发现过程持续演进
- 每周根据新信息更新
- 反映团队当前的理解

## 如何构建

### Step 1: 定义期望结果
```markdown
问自己：
- 我们要改变什么行为？
- 如何衡量成功？
- 这个结果如何与公司目标对齐？
```

### Step 2: 发现机会
- 通过用户访谈收集故事
- 从故事中提取痛点和需求
- 用用户的语言记录机会

### Step 3: 探索解决方案
- 为每个机会头脑风暴多个解决方案
- 不要过早收敛
- 考虑不同规模的解决方案

### Step 4: 设计假设测试
- 识别每个解决方案的核心假设
- 设计快速实验验证假设
- 根据结果调整方向

## 常见错误

1. **跳过机会层**
   - 直接从结果跳到解决方案
   - 错失真正重要的用户问题

2. **单线程思考**
   - 只考虑一个选项
   - 无法做出好的比较决策

3. **静态使用**
   - 创建一次就不更新
   - 没有反映最新的发现

4. **解决方案伪装成机会**
   - "用户需要一个仪表盘"（解决方案）
   - vs "用户难以了解他们的进度"（机会）

|------|---------------|

| Experience Map | 输入：帮助组织和结构化机会 |
| Assumption Testing | 输出：验证解决方案的方法 |
| Story Mapping | 配合：详细规划选定的解决方案 |
