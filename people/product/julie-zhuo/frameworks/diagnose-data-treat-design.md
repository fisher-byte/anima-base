# Diagnose with Data, Treat with Design

> "Diagnose with data, treat with design." — Julie Zhuo

## Overview

This framework separates the **measurement** phase from the **solutioning** phase in product development. It's Julie Zhuo's approach to balancing intuition with experimentation when building great products.

## The Framework

### Phase 1: Diagnose with Data 🔍

**Purpose**: Understand *what* is happening and *where* problems exist.

**Activities**:
- Analyze quantitative metrics (engagement, retention, conversion)
- Conduct user research to understand behaviors
- Identify patterns and anomalies in user data
- Measure current state against benchmarks

**Key Questions**:
- What does the data tell us about user behavior?
- Where are users dropping off or struggling?
- What patterns do we see across different segments?
- How does our current performance compare to goals?

**Output**: Clear problem definition backed by evidence

### Phase 2: Treat with Design 🎨

**Purpose**: Create solutions that address diagnosed problems.

**Activities**:
- Brainstorm multiple solution approaches
- Prototype and test with users
- Iterate based on qualitative feedback
- Trust design intuition for the "how"

**Key Questions**:
- What's the user's underlying need or pain point?
- What solutions could address this problem?
- How should the experience *feel* to users?
- What's the simplest version we can test?

**Output**: Designed solutions ready for validation

## Why It Works

| Approach | Strength | Weakness |
|----------|----------|----------|
| Data-only | Objective, scalable | Can't tell you *what* to build |
| Intuition-only | Creative, human-centered | Can be wrong, hard to scale |
| **Combined** | **Finds real problems, creates better solutions** | **Requires both skills** |

## Application Examples

### Example 1: Onboarding Flow

**Diagnose (Data)**:
- 40% of users drop off at step 3 of onboarding
- Users who complete onboarding have 3x higher retention
- Mobile users drop off more than desktop

**Treat (Design)**:
- Redesign step 3 to be simpler and more visual
- Add progress indicator to show users how close they are
- Create mobile-optimized version with larger touch targets

### Example 2: Feature Adoption

**Diagnose (Data)**:
- New feature has only 5% discovery rate
- Users who find it use it repeatedly
- Search logs show users looking for related functionality

**Treat (Design)**:
- Add contextual prompts when users perform related actions
- Create guided tour for new users
- Surface feature in relevant empty states

## Common Mistakes

### ❌ Treating with Data

- A/B testing tiny variations without understanding user needs
- Optimizing metrics without considering user experience
- Letting data dictate creative decisions

### ❌ Diagnosing with Design

- Assuming you know the problem without validation
- Building solutions before understanding the problem
- Ignoring contradictory data because "it feels right"

## When to Trust Intuition

Julie acknowledges that data isn't everything:

> "Sometimes you need to make bets that data can't validate yet."

**Trust design intuition when**:
- Creating something genuinely new (no data yet)
- Optimizing for delight vs. efficiency
- Making brand or emotional experience decisions
- Data is noisy or contradictory

## Integration with AI Products

In the AI era, this framework applies to:

1. **Diagnosing AI performance** with metrics (accuracy, latency, user satisfaction)
2. **Treating AI experience** with prompts, interfaces, and error handling design

Julie notes that managing AI agents follows similar patterns to managing humans—you need clear objectives (data) and thoughtful execution (design).
