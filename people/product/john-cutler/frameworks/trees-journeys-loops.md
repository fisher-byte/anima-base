# Trees, Journeys, and Loops

## 框架概述

在进行数百次 North Star workshops 后，John Cutler 发现团队（和职能部门）往往偏向于其中一种视角：**能力视角（Trees）**、**旅程视角（Journeys）** 或 **飞轮视角（Loops）**。

通过有技巧地使用这三种视角，可以建立更有韧性的跨职能共享语言，避免陷入"职能模型孤岛陷阱"。

> "不要让一个职能主导！避免职能陷阱！"
> — John Cutler

---

## 三种模型类型

### 1. 树型（Trees / Capability View）

**特点**:
- 层级结构
- 分解和组合
- 能力和组件导向

**典型应用**:
- 功能树
- 指标树（Driver Trees）
- 组织架构
- OKR 层级

**优势**:
- 清晰的分解
- 便于分配责任
- 易于理解层级关系

**局限**:
- 可能忽略横向连接
- 容易形成孤岛
- 不反映动态流程

**偏好人群**: 工程师、项目经理、财务

---

### 2. 旅程型（Journeys / Flow View）

**特点**:
- 线性流程
- 阶段和转化
- 用户体验导向

**典型应用**:
- 用户旅程地图
- 漏斗分析
- 服务蓝图
- AARRR 框架

**优势**:
- 反映用户真实体验
- 发现摩擦点和机会
- 跨触点的视角

**局限**:
- 可能过于简化非线性行为
- 忽略循环和反馈
- 难以反映多路径

**偏好人群**: 设计师、UX 研究员、增长团队

---

### 3. 循环型（Loops / Flywheel View）

**特点**:
- 循环反馈
- 增强效应
- 系统动态

**典型应用**:
- 增长飞轮
- 反馈循环
- 网络效应模型
- 系统动力学

**优势**:
- 反映复利效应
- 发现增强循环
- 理解系统行为

**局限**:
- 可能过于抽象
- 难以分配具体责任
- 不反映线性流程

**偏好人群**: 战略师、增长专家、系统思维者

## 职能偏好陷阱

John Cutler 观察到不同职能有不同的默认视角：

| 职能 | 偏好视角 | 典型表现 |
|------|----------|----------|
| **工程** | 树型 | "让我们分解成组件" |
| **设计** | 旅程型 | "用户的旅程是什么？" |
| **增长** | 循环型 | "如何创造飞轮效应？" |
| **产品** | 混合 | 取决于背景和经验 |

### 陷阱表现

当一个职能的视角主导时：

- **工程主导**: 功能被很好分解，但用户体验碎片化
- **设计主导**: 旅程清晰，但技术实现困难
- **增长主导**: 飞轮设计精妙，但缺乏执行细节

## 框架应用

### 整合三种视角

最佳实践是**同时使用三种视角**，而不是选择一种：

```
          ┌─────────────┐
          │   LOOPS     │  ← 战略层：增长飞轮
          │  (Flywheel) │
          └─────────────┘
                ↓
          ┌─────────────┐
          │  JOURNEYS   │  ← 体验层：用户旅程
          │   (Flow)    │
          └─────────────┘
                ↓
          ┌─────────────┐
          │   TREES     │  ← 执行层：能力分解
          │(Capability) │
          └─────────────┘
```

### 对话框架

在跨职能讨论中：

1. **从 Loops 开始**: "我们的增长引擎是什么？"
2. **落到 Journeys**: "用户如何体验这个飞轮？"
3. **细化到 Trees**: "我们需要什么能力来实现？"

### North Star 与三视角

将 North Star Framework 与三视角结合：

- **North Star Metric**: 可以是 Loop 的核心指标
- **Input Metrics**: 来自 Journey 的关键转化点
- **Work & Experiments**: 对应 Tree 中的能力建设

## 实际案例

### 电商平台的三视角

**Loops（飞轮）**:
```
更多商品 → 更多买家 → 更多卖家 → 更多商品 (循环)
```

**Journeys（旅程）**:
```
发现 → 浏览 → 加购 → 结算 → 收货 → 复购
```

**Trees（能力）**:
```
平台
├── 搜索能力
│   ├── 推荐算法
│   └── 筛选系统
├── 交易能力
│   ├── 支付系统
│   └── 物流跟踪
└── 信任能力
    ├── 评价系统
    └── 客服系统
```

## 实施建议

### 在 Workshop 中

1. **分组练习**: 让不同职能用自己偏好的视角建模
2. **交叉分享**: 展示不同视角如何互补
3. **整合讨论**: 找到三种视角的连接点

### 在日常工作中

1. **检查盲点**: "我们是否只用了一种视角？"
2. **邀请不同声音**: 让不同背景的人参与讨论
3. **可视化三视角**: 在文档中同时展示三种模型

### 避免的行为

- ❌ 让一个职能的视角主导所有讨论
- ❌ 认为某种视角"更正确"
- ❌ 忽略其他视角带来的洞察

---

## 与其他框架的关系

- **North Star Framework**: 三视角帮助设计更完整的 North Star 体系
- **OKR**: Trees 是 OKR 的自然结构，但需要 Loops 来检验长期效应
- **User Story Mapping**: 本质是 Journeys 视角的应用
