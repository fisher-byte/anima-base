# 产品团队的OKR (OKRs for Product Teams)

## 框架概述

OKR（Objectives and Key Results）是目标与关键结果的缩写，由 Intel 的 Andy Grove 发明，John Doerr 推广到 Google 等公司。Marty Cagan 将 OKR 作为产品团队从产出导向转向成果导向的核心工具。

## 为什么产品团队需要 OKR

### 传统路线图的问题

- 功能列表而非业务成果
- 团队不知道"为什么"
- 缺乏衡量标准
- 变成承诺而非假设

### OKR 的价值

- 明确业务成果目标
- 赋能团队找解决方案
- 对齐个人与组织目标
- 聚焦最重要的事

## OKR 基本结构

### Objective (目标)

**定义**: 你想要实现什么

**特征**:
- 定性的、鼓舞人心的
- 有意义、有挑战
- 团队可控
- 不是功能描述

**好的 Objective 示例**:
- "显著提升新用户首周留存"
- "成为中小企业最信赖的支付方案"
- "让用户更容易发现相关内容"

**坏的 Objective 示例**:
- "发布推荐系统v2"（功能，不是成果）
- "完成10个用户访谈"（任务，不是成果）
- "提升NPS"（太模糊，不可执行）

### Key Results (关键结果)

**定义**: 如何知道你达成了目标

**特征**:
- 定量的、可衡量的
- 有具体数字
- 2-4个KR per O
- 是领先指标或结果指标

**好的 Key Results 示例**:
- "首周留存率从30%提升到45%"
- "月活跃用户数达到100万"
- "用户任务完成率提升20%"

**坏的 Key Results 示例**:
- "发布3个新功能"（产出，不是成果）
- "用户满意"（不可衡量）
- "做得更好"（无意义）

## 产品团队 OKR 示例

### 示例1: 增长团队

```
Objective: 大幅提升新用户转化
Key Results:
- KR1: 注册到首次付费转化率从5%提升到10%
- KR2: 用户激活时间从7天缩短到3天
- KR3: 首月流失率从40%降低到25%
```

### 示例2: 核心体验团队

```
Objective: 让用户更容易完成核心任务
Key Results:
- KR1: 任务完成率从60%提升到85%
- KR2: 平均任务时间从5分钟缩短到2分钟
- KR3: 用户求助率从15%降低到5%
```

### 示例3: 平台团队

```
Objective: 提升平台可靠性和开发效率
Key Results:
- KR1: 系统可用性从99.5%提升到99.95%
- KR2: 部署频率从每周1次提升到每天3次
- KR3: 关键Bug修复时间从24小时缩短到4小时
```

## OKR vs 功能路线图

| 维度 | OKR | 功能路线图 |
|-----|-----|-----------|
| **内容** | 业务成果 | 功能列表 |
| **团队角色** | 发现解决方案 | 实现指定功能 |
| **灵活性** | 方法可调整 | 功能固定 |
| **衡量** | KR达成度 | 按时发布 |
| **学习** | 验证假设 | 执行计划 |

## OKR 设定原则

### 1. 成果导向

> "OKRs should be about outcomes, not outputs."

**问自己**:
- 这是结果还是活动？
- 如果我们做了这些，业务会有什么改变？
- 用户会有什么不同？

### 2. 团队可控

- KR 应该是团队能影响的
- 不应该完全依赖外部因素
- 给团队发现解决方案的空间

### 3. 有挑战性

- 太容易达成没有意义
- 60-70% 达成率是理想的
- 促使团队思考创新方案

### 4. 对齐但不级联

- 与公司目标对齐
- 但不是机械地分解
- 每个层级有自主权

### 5. 少而精

- 每个团队1-3个Objective
- 每个O有2-4个KR
- 聚焦最重要的

## OKR 常见问题

### Q: OKR应该多少达成算成功？

A: 如果团队经常100%达成，说明目标设得太低。60-70%达成率是健康的。

### Q: OKR和绩效评估挂钩吗？

A: Marty Cagan 和 John Doerr 都建议不要直接挂钩，否则团队会设置保守目标。

### Q: KR可以是功能吗？

A: 不可以。KR应该是业务指标，功能是达成KR的手段，不是KR本身。

### Q: 多久设一次OKR？

A: 通常是季度，但可以根据业务节奏调整。

## OKR 反模式

### ❌ 功能伪装成KR

```
Bad:
O: 提升搜索体验
KR: 发布搜索v2

Good:
O: 帮助用户更快找到所需内容
KR: 搜索点击率从10%提升到25%
```

### ❌ KR太多

- 超过4个KR意味着不聚焦
- 团队会分散精力
- 保持2-3个核心KR

### ❌ 指标游戏

- 优化指标而非真正价值
- 需要结合定性判断
- 定期审视指标是否仍然相关

### ❌ 设完就忘

- OKR需要定期review
- 通常每周或每两周check-in
- 根据学习调整策略

## Marty Cagan 原话

> "The key result is the outcome you're trying to achieve, not the output. Building a feature is not a key result – that's an output. The impact of the feature is the key result."

> "OKRs are what allow teams to move from being given features to build, to being given problems to solve."
